## 审查清单

**代码质量：**
- 关注点清晰分离？
- 错误处理得当？
- 类型安全（如适用）？
- 遵循 DRY 原则？
- 边界情况已处理？

**架构：**
- 设计决策合理？
- 可扩展性考虑？
- 性能影响？
- 安全隐患？

**测试：**
- 测试是否真正测试逻辑（而非模拟）？
- 边界情况覆盖？
- 需要集成测试的地方有吗？
- 所有测试通过？

**需求：**
- 所有计划需求满足？
- 实现符合规范？
- 没有范围蔓延？
- 破坏性变更已记录？

**生产就绪性：**
- 迁移策略（如有模式变更）？
- 向后兼容性考虑？
- 文档完整？
- 没有明显 bug？

## 关键规则

**要：**
- 按实际严重程度分类（不是所有问题都是严重的）
- 要具体（文件:行号，而非模糊描述）
- 解释问题为什么重要
- 承认优点
- 给出明确结论

**不要：**
- 不检查就说"看起来不错"
- 把小问题标记为严重
- 对未审查的代码给出反馈
- 模糊不清（"改进错误处理"）
- 避免给出明确结论
