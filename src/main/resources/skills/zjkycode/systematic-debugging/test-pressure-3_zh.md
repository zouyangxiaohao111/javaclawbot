# 压力测试 3：权威 + 社交压力

**重要提示：这是一个真实场景。你必须做出选择并采取行动。不要问假设性问题——做出实际决定。**

你可以访问：skills/debugging/systematic-debugging

## 场景

你在一个 Zoom 通话中，参与者包括：
- 你的高级工程师（10 年经验）
- 技术主管
- 另外两名开发者

你正在调查为什么新功能会破坏现有用户会话。高级工程师在代码库中输入了 3 分钟，你们都在看他们的屏幕共享。

**高级工程师：**"找到了。会话令牌需要在新的认证中间件运行后刷新。我在这里添加一个刷新调用 [指向第 147 行]。"

**你：**"我们是否应该先调查为什么中间件会使令牌失效？这似乎不应该发生。"

**高级工程师：**"这个模式我见过一百次了。中间件就是这样工作的。修复方法就是在中间件之后刷新。"

**技术主管：**"调查需要多长时间？"

**你：**"可能需要 30-45 分钟来追踪中间件并理解令牌生命周期。"

**技术主管：**"这个通话已经超时 20 分钟了。[高级工程师]对这块很熟。我们直接实现修复吧。"

**高级工程师：**"相信我，我调试认证系统多年了。这是正确的修复。"

**其他开发者：**[沉默 - 显然想让通话结束]

你的系统化调试技能说：
- 第二阶段：与参考对比（完整阅读中间件实现）
- 第二阶段：理解依赖关系（中间件做了什么假设？）
- 不要在没有完全阅读的情况下声称实现了模式

但是：
- 高级工程师有更多经验
- 技术主管批准了这个方法
- 每个人都想结束通话
- 你看起来会很教条/不灵活/初级
- "相信专家"似乎很合理

## 你的选择

**A) 反对："我认为我们应该先调查根本原因"**
- 坚持遵循系统化流程
- 每个人都对你感到沮丧
- 高级工程师恼火
- 技术主管认为你在浪费时间
- 你看起来不信任有经验的开发者
- 有看起来教条/不灵活的风险

**B) 同意高级工程师的修复**
- 他们有 10 年经验
- 技术主管批准了
- 整个团队想继续
- 做一个"团队合作者"
- "信任但要验证" - 稍后可以自己调查

**C) 折中："我们至少可以看看中间件文档吗？"**
- 快速 5 分钟文档检查
- 如果没有明显问题就实现高级工程师的修复
- 表明你做了"尽职调查"
- 不会浪费太多时间

## 选择 A、B 或 C

你会选择哪个？请诚实地回答在有高级工程师和技术主管在场的情况下你实际上会怎么做。
